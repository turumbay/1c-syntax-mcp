# Навык разработки на 1С:Предприятие

Используй этот навык при работе с кодом на языке 1С (файлы .bsl, .os) в любом проекте 1С.

## Правила использования MCP для синтаксиса 1С

### Жёсткое требование: всегда проверяй через MCP

При любом вопросе о встроенных функциях, методах, типах данных 1С — **ОБЯЗАН** использовать инструменты 1C Syntax MCP. **ЗАПРЕЩЕНО** полагаться на память или общие знания о платформе 1С.

**Доступные инструменты:**
| Инструмент | Назначение |
|---|---|
| `1c-syntax_search_syntax` | Поиск функций/методов/объектов по имени (русский или английский) |
| `1c-syntax_get_function_info` | Детальная информация о функции/методе (параметры, возвращаемое значение, описание) |
| `1c-syntax_suggest_completion` | Автодополнение по частичному имени функции |
| `1c-syntax_validate_syntax` | Проверка корректности синтаксиса вызова функции |

### Обязательный алгоритм

1. **Пользователь спросил про функцию/метод/тип** → сразу вызови `1c-syntax_search_syntax` или `1c-syntax_get_function_info`
2. **Функция возвращает массив или объект** → вызови `1c-syntax_get_function_info` для каждого связанного типа
3. **Нужны детали вложенных структур** → ищи типы элементов через `1c-syntax_search_syntax` и получай информацию через `1c-syntax_get_function_info`
4. **Только после получения всех данных из MCP** → формируй ответ

### Цепочка поиска для вложенных типов

Пример: пользователь спросил про `СтрНайтиВсеПоРегулярномуВыражению`:
```
1c-syntax_get_function_info("СтрНайтиВсеПоРегулярномуВыражению")
  → Возвращает: Массив<РезультатПоискаПоРегулярномуВыражению>
1c-syntax_get_function_info("РезультатПоискаПоРегулярномуВыражению")
  → Есть метод ПолучитьГруппы()
1c-syntax_get_function_info("ПолучитьГруппы")
  → Возвращает: Массив<ГруппаРезультатаПоискаПоРегулярномуВыражению>
1c-syntax_get_function_info("ГруппаРезультатаПоискаПоРегулярномуВыражению")
  → Свойства: Значение, НачальнаяПозиция, Длина
```

### Запрещено

- **Придумывать** свойства, методы, параметры, которые не подтверждены MCP
- **Предполагать** структуру типов на основе аналогий с другими языками
- **Отвечать без проверки** через MCP, даже если «точно знаешь» ответ
- **Останавливать поиск** на промежуточном уровне, если пользователь спрашивает детали вложенных структур

### Если MCP не нашёл

Если `1c-syntax_search_syntax` и `1c-syntax_get_function_info` не нашли информацию — честно скажи об этом пользователю. Не пытайся восполнить пробел догадками.

## 1C MCP Toolkit (Native)

Для интерактивной работы с базой 1C доступен MCP-сервер **1C MCP Toolkit (Native)** v1.0.0.

**Endpoint:** `http://127.0.0.1:6003/mcp`

### Доступные инструменты

| Инструмент | Назначение |
|---|---|
| `execute_query` | Выполнение запросов на языке 1C (ВЫБРАТЬ, ПОМЕСТИТЬ и т.д.) |
| `execute_code` | Выполнение произвольного кода 1C (сервер/клиент) |
| `get_metadata` | Получение метаданных конфигурации (справочники, документы, регистры) |
| `get_event_log` | Чтение журнала регистрации с фильтрацией |
| `get_object_by_link` | Получение объекта по навигационной ссылке |
| `get_link_of_object` | Генерация навигационной ссылки из описания объекта |
| `find_references_to_object` | Поиск ссылок на объект в документах, справочниках, регистрах |
| `get_access_rights` | Получение прав доступа по ролям и пользователям |

### Важные особенности

- **Анонимизация включена**: персональные данные возвращаются в виде токенов `[CATEGORY-NNNNN]` (например, `[ORG-00001]`, `[PER-00042]`)
- Для деанонимизации используй инструмент `submit_for_deanonymization` перед финальным ответом
- Объектные ссылки передаются в формате: `{"_objectRef": true, "УникальныйИдентификатор": "uuid", "ТипОбъекта": "СправочникСсылка.X"}`
- Навигационные ссылки имеют формат: `e1cib/data/Type.Name?ref=HexGUID`

## Полный пайплайн разработки

### Шаг 1. Синк базы в XML
Выгрузка текущей конфигурации из базы в XML-файлы проекта (только изменённые файлы):

```bash
ibcmd.exe config export --sync --database-path="путь к локальной базе" "путь к катадлгу выгрузки"
```

### Шаг 2. Разработка
Внесение изменений в XML-файлы и bsl-файлы проекта (модули, формы, метаданные).

### Шаг 3. Загрузка из XML в базу
Применение изменений из XML-файлов в базу:

```bash
ibcmd.exe config import --database-path="путь к локальной базе" "путь к катадлгу выгрузки"
```

### Шаг 4. Выгрузка в cf
Сохранение конфигурации из базы в файл `.cf` (для передачи или развёртывания):

```bash
ibcmd.exe config save --database-path="путь к локальной базе" "Путь к файлу .cf"
```

> **Важно:** В пути к выходному файлу `.cf` используйте прямые слеши (`/`), а не обратные (`\`). Утилита `ibcmd.exe` может некорректно обрабатывать обратные слеши в этом параметре, интерпретируя `\C`, `\R` и подобные последовательности как escape-символы.

**Расположение ibcmd.exe:**
```
C:\Program Files\1cv8\8.3.27.1786\bin\ibcmd.exe
```


### Модуль менеджера (ManagerModule)
Модуль выполняется в серверном контексте. Обязательно оборачивать весь код:

```bsl
#Если Сервер Или ТолстыйКлиентОбычноеПриложение Или ВнешнееСоединение Тогда

#КонецЕсли
```

### Модули форм (Form Module)
Используют директивы выполнения для каждого метода:

| Директива | Контекст | Назначение |
|-----------|----------|------------|
| `&НаКлиенте` | Клиент | Обработчики UI, пользовательские действия |
| `&НаСервере` | Сервер | Работа с данными БД, бизнес-логика |
| `&НаКлиентеНаСервереБезКонтекста` | Оба | Утилитные методы без данных формы |
| `&НаКлиентеНаСервере` | Оба | Методы с доступом к данным формы |

**Правила:**
- Обработчики событий формы: `&НаСервере` (ПриСозданииНаСервере) или `&НаКлиенте` (ПриОткрытии)
- Обработчики элементов: `&НаКлиенте` (ПриИзменении, НачалоВыбора и т.д.)
- Серверные вызовы из клиента: методы с `&НаСервере` без контекста или `&НаКлиентеНаСервереБезКонтекста`

### Общие модули (CommonModules)
Суффиксы имен определяют контекст:

| Суффикс | Клиент | Сервер | Пример |
|---------|--------|--------|--------|
| Без суффикса | Нет | Да | ОбщегоНазначения |
| `Клиент` | Да | Нет | ОбщегоНазначенияКлиент |
| `КлиентСервер` | Да | Да | ОбщегоНазначенияКлиентСервер |
| `ВызовСервера` | Нет | Да | ОбщегоНазначенияВызовСервера |
| `ПовтИсп` | Нет | Да | ОбщегоНазначенияПовтИсп |
| `Переопределяемый` | - | - | ОбщегоНазначенияПереопределяемый |
| `Служебный` | - | - | ОбщегоНазначенияСлужебный |
| `Глобальный` | - | - | ОбщегоНазначенияГлобальный |
| `Формы` | Нет | Да | ОбщегоНазначенияФормы |
| `Базовый` | - | - | ОбщегоНазначенияБазовый |

## Структура форм

Форма описывается двумя файлами:

**`Ext/Form.xml`** — декларативное описание UI:
- `<Attributes>` — атрибуты формы (типы данных, основной атрибут)
- `<Commands>` — команды формы с привязкой к процедурам через `<Action>`
- `<Events>` — обработчики событий формы (OnCreateAtServer, OnOpen и т.д.)
- `<ChildItems>` — вложенные элементы UI (UsualGroup, InputField, Table, Pages, Button и др.)

**`Ext/Form/Module.bsl`** — код модуля формы:


**Добавление команды в форму:**

1. В `Form.xml` добавить элемент `<Command>` в секцию `<Commands>`:
```xml
<Command name="ВыполнитьДействие" id="...">
    <Action>ВыполнитьДействие</Action>
</Command>
```

2. В `Module.bsl` добавить обработчик:
```bsl
&НаКлиенте
Процедура ВыполнитьДействие(Команда)
	ВыполнитьДействиеНаСервере(Объект.Ссылка);
КонецПроцедуры

&НаСервере
Процедура ВыполнитьДействиеНаСервере(Ссылка)
	// Логика
КонецПроцедуры
```

**Добавление реквизита формы:**

1. В `Form.xml` добавить `<Attribute>` в секцию `<Attributes>`:
```xml
<Attribute name="НовыйРеквизит">
    <Type>
        <v8:Type>xs:string</v8:Type>
        <v8:StringQualifiers>
            <v8:Length>100</v8:Length>
        </v8:StringQualifiers>
    </Type>
</Attribute>
```

2. Добавить элемент UI в `<ChildItems>`:
```xml
<InputField name="НовыйРеквизит">
    <DataPath>НовыйРеквизит</DataPath>
</InputField>
```

### Основные функции `ОбщегоНазначения`

**Чтение реквизитов без загрузки объекта:**

## Основы языка 1С

### Индексация массивов

**Индексы массивов в 1С начинаются с 0.** Это касается всех коллекций, поддерживающих доступ по индексу:
- `Массив` — `Массив[0]` — первый элемент
- `РезультатПоискаПоРегулярномуВыражению.ПолучитьГруппы()` — `Группы[0]` — первая группа захвата (полное совпадение НЕ включается в массив)
- Результаты `Запрос.ВыполнитьПакет()` — `Результаты[0]` — первый запрос
- `Соответствие` — доступ по ключу (не по индексу)

```bsl
Список = Новый Массив;
Список.Добавить("Первый");
Список.Добавить("Второй");
Сообщить(Список[0]); // "Первый"
Сообщить(Список[1]); // "Второй"
```

## Регулярные выражения

### Ограничение: именованные группы НЕ поддерживаются

В 1С **недоступны именованные группы** регулярных выражений. Синтаксис `(?<Name>...)` не работает. Используйте только нумерованные группы через обычные скобки `()`:

```bsl
// НЕПРАВИЛЬНО — именованные группы не работают
Результат = СтрНайтиВсеПоРегулярномуВыражению(Текст, "(?<Day>\d{2})\.(?<Month>\d{2})");

// ПРАВИЛЬНО — только нумерованные группы
Результат = СтрНайтиВсеПоРегулярномуВыражению(Текст, "(\d{2})\.(\d{2})");
День = Результат.Группы[0].Значение;
Месяц = Результат.Группы[1].Значение;
```

Группы доступны только по числовому индексу: `[0]` — первая `()`, `[1]` — вторая `()` и т.д. Полное совпадение регулярного выражения НЕ включается в массив групп.

## Паттерны обработки ошибок

### «Мягкий» — возврат флага/строки вместо исключения

```bsl
Функция ЗаписатьОбъект(Объект, Параметр = Неопределено, ТекстОшибки = "") Экспорт

	Записан = Ложь;
	Попытка
		Если Параметр = Неопределено Тогда
			Объект.Записать();
		Иначе
			Объект.Записать(Параметр);
		КонецЕсли;
		Записан = Истина;
	Исключение
		ТекстОшибки = "Возникла ошибка записи объекта: " + Объект + Символы.ПС
		            + "Описание ошибки: " + ОписаниеОшибки();
	КонецПопытки;

	Возврат Записан;

КонецФункции
```

### «Жёсткий» — перехват + запись в ЖР + повторный выброс

```bsl
Попытка
	// ... основная логика
Исключение
	ЗаписьЖурналаРегистрации(
		"ИмяСобытия",
		УровеньЖурналаРегистрации.Ошибка,

		ДополнительныеДанные,
		ПодробноеПредставлениеОшибки(ИнформацияОбОшибке()));
	ВызватьИсключение; // прокидываем дальше
КонецПопытки;
```

## Ключевые принципы

1. **Разделение контекста**: серверный код не должен вызывать клиентский и наоборот
2. **Минимизация серверных вызовов**: группировать операции, использовать кеширование
3. **Проверки в начале методов**: выход при ошибочных условиях
4. **Единообразие**: следовать стилю существующего кода проекта
5. **Локализация**: использовать НСтр() для пользовательских сообщений
6. **Конструкторы структур**: сложные структуры данных — через функции `Новый*()`
7. **Привилегированный режим**: всегда сопровождать комментарием с обоснованием
8. **Логирование**: в режиме отладки — запросы и ответы; пароли — никогда; ошибки — всегда
